--------------------------------------------------------------------------------
RF01 – Solicitar Agendamento
--------------------------------------------------------------------------------

Descrição:
Permitir que o paciente realize uma solicitação de agendamento informando seus
dados básicos e o tipo de atendimento desejado.

História de Usuário:
Eu, Rafael (Paciente), desejo conseguir preencher um formulário com meus dados
e o tipo de exame para formalizar o pedido de atendimento.

Critérios de Aceitação:
1. O sistema deve validar campos obrigatórios (Nome, CPF e contato) antes do envio.
2. O sistema deve permitir a seleção do atendimento via menu dinâmico.
3. O sistema deve gerar um número de protocolo ao finalizar o registro.
4. O sistema deve impedir que o paciente realize uma nova solicitação idêntica
   (mesmo CPF e mesmo tipo de exame) caso já exista um pedido com status
   "Pendente" ou "Em análise", exibindo uma mensagem de aviso.
5. O sistema deve exigir o upload de uma foto ou PDF do encaminhamento
   médico/pedido de exame antes de permitir a finalização da solicitação.
6. Após a geração do número de protocolo, o sistema deve exibir uma mensagem
   de confirmação na tela e fornecer instruções claras sobre o prazo médio de
   resposta para a triagem administrativa.

RF02 – Informar Cidade de Origem
--------------------------------------------------------------------------------

Descrição:
Permitir que o paciente selecione a cidade de origem no momento da solicitação.

História de Usuário:
Eu, Lúcia (Paciente), desejo conseguir indicar minha cidade para que o hospital
consiga priorizar minha vaga por região.

Critérios de Aceitação:
1. O campo de seleção de cidade deve ser obrigatório no formulário.
2. O sistema deve oferecer uma lista de cidades integradas para a busca rápida.
3. O sistema deve vincular a cidade ao registro para fins de relatório estatístico.
4. O sistema deve sugerir automaticamente as cidades de Sergipe ao detectar o
   prefixo (79) no campo de contato, facilitando o preenchimento para residentes locais.
5. Ao selecionar uma cidade fora de Sergipe, o sistema deve exigir obrigatoriamente
   a seleção do Estado (UF) para evitar inconsistências nos relatórios geográficos.
6. O sistema deve classificar internamente a cidade selecionada em uma das regiões
   de saúde pré-definidas (ex: Região de Lagarto, Região de Aracaju), permitindo que
   a triagem visualize a demanda por polos regionais.

RF04 – Visualizar Resumo da Solicitação
--------------------------------------------------------------------------------

Descrição:
Exibir ao paciente um resumo da solicitação antes da confirmação.

História de Usuário:
Eu, Marcos (Acompanhante), desejo ver e revisar todos os dados da solicitação
antes de confirmar para evitar erros no agendamento.

Critérios de Aceitação:
1. O sistema deve exibir uma tela de revisão com todos os dados preenchidos.
2. O sistema deve disponibilizar um botão "Editar" para correções rápidas na mesma tela.
3. O envio definitivo ao banco de dados só deve ocorrer após a confirmação final nesta tela.
4. O resumo deve exibir o nome e permitir a abertura (preview) de quaisquer documentos

5. A tela de resumo deve incluir um "checkbox" obrigatório onde o usuário declara que

   dados para fins de atendimento.
6. O sistema deve exibir uma sinalização visual clara se a solicitação foi identificada
   como "Prioridade Legal", permitindo que o usuário valide se essa condição foi
   reconhecida pelo sistema.

RF05 – Registrar Solicitação
--------------------------------------------------------------------------------

Descrição:
Registrar a solicitação realizada pelo paciente em uma fila de atendimento.

História de Usuário:
Eu, Cláudio (Analista de dados), prefiro que o sistema salve automaticamente os
dados enviados na fila de espera para que a administração processe o pedido.

Critérios de Aceitação:
1. O sistema deve gravar o registro com carimbo de data e hora.
2. O sistema deve definir o status inicial da solicitação como "Pendente".
3. O sistema deve disponibilizar o registro no módulo institucional em tempo real.
4. Além do número de protocolo sequencial, o sistema deve gerar uma chave interna
   única no banco de dados para garantir a integridade referencial em futuras
   integrações e relatórios estatísticos.
5. O sistema deve armazenar, junto ao registro, informações técnicas da origem da
   solicitação (como o endereço IP ou o ID da sessão), permitindo que o analista
   identifique padrões de uso ou possíveis envios automatizados indevidos.
6. No momento exato do registro bem-sucedido, o sistema deve disparar uma
   notificação interna para o dashboard dos recepcionistas e um e-mail de confirmação
   automática para o paciente, contendo o resumo registrado e o protocolo.

RF06 – Acompanhar Status
--------------------------------------------------------------------------------

Descrição:
Permitir que o paciente acompanhe o status da sua solicitação.

História de Usuário:
Eu, Beatriz (Paciente), desejo conseguir consultar a situação atual do meu pedido
para saber se ele já foi aprovado ou agendado.

Critérios de Aceitação:
1. A consulta deve estar disponível via número de protocolo ou login de usuário.
2. O sistema deve exibir claramente os status: Pendente, Confirmado ou Cancelado.
3. O sistema deve manter o histórico de mudanças de status visível para o usuário.
4. Caso o status seja alterado para "Cancelado", o sistema deve exibir obrigatoriamente
   o motivo registrado pela administração (ex: documentação ilegível, falta de vaga no
   período), orientando a paciente sobre os próximos passos.
5. Ao atingir o status "Confirmado", o sistema deve exibir automaticamente a data,
   o horário, o nome do médico/especialidade e o local (bloco/sala) do atendimento.
6. Caso o status seja "Em Análise", o sistema deve exibir uma estimativa de prazo
   de resposta, atualizada dinamicamente com base na média de triagem do último
   mês, para gerenciar a expectativa do paciente.

RF07 – Enviar Notificações por Email
--------------------------------------------------------------------------------

Descrição:
Enviar notificações por e-mail ao paciente informando atualizações no status do
atendimento.

História de Usuário:
A automação de mensagens por email deve enviar alertas automáticos para avisar
o paciente sobre mudanças no status sem que ele precise abrir o site.

Critérios de Aceitação:
1. O sistema deve disparar mensagens automaticamente a cada atualização de
   status no banco de dados.
2. O conteúdo da mensagem deve ser claro e conter o link para consulta detalhada.
3. O sistema deve registrar um log de envio para confirmar que a notificação foi disparada.
4. O sistema deve utilizar templates de e-mail específicos para cada situação (ex:
   um texto acolhedor para "Confirmado" e instruções técnicas de correção de documentos
   para "Pendente"), garantindo que a linguagem seja adequada à etapa do processo.
5. Caso o servidor de e-mail retorne um erro de entrega (bounce), o sistema deve
   sinalizar essa falha no dashboard administrativo, permitindo que a recepção tente
   contato por outro meio (como o telefone registrado).
6. O corpo do e-mail não deve conter dados sensíveis de saúde ou diagnósticos
   detalhados; a mensagem deve apenas informar a alteração do status e direcionar o
   usuário para o ambiente logado seguro através do link de consulta.

RF41 – Coletar Feedback do Paciente
--------------------------------------------------------------------------------

Descrição:
Permitir que o paciente informe seu nível de satisfação com o serviço, a partir de
uma escala numérica e comentários qualitativos.

História de Usuário:
Eu, Carla (Paciente), desejo receber uma pesquisa simples e rápida após meu
agendamento e atendimento, para que eu possa avaliar a experiência e enviar
sugestões ou críticas para a instituição.

Critérios de Aceitação:
1. O sistema deve disparar automaticamente a solicitação de feedback (NPS) 24 horas
   após o agendamento atingir o status "Confirmado" e, novamente, 24 horas após o
   status ser alterado para "Concluído" (pós-atendimento).
2. A pesquisa deve ser composta por uma pergunta central de satisfação em escala de 0 a 10
   e por um campo de texto livre obrigatório para comentário.
3. O sistema deve registrar cada resposta de feedback em um banco de dados dedicado,
   vinculando-a ao número de protocolo do agendamento e registrando a data/hora da
   resposta (timestamp).
4. O link da pesquisa de feedback deve ser único por protocolo de agendamento e
   temporário (com validade de 7 dias após o envio), garantindo que apenas o paciente
   destinatário possa responder.
5. O sistema deve garantir que o paciente possa responder à pesquisa apenas uma vez
   por agendamento (evitando múltiplas submissões) e que o link da pesquisa expire
   após 7 dias da data de envio.
6. O sistema deve armazenar, junto ao feedback, o endereço IP e o ID da sessão do
   usuário, para fins de auditoria e para identificar possíveis padrões de fraude ou
   uso indevido.

RF49 – Gerar Credencial de Acesso com QR Code
--------------------------------------------------------------------------------

Descrição:
Permitir a geração de uma credencial digital para cada agendamento confirmado,
contendo um QR Code único e criptografado. Este código deve funcionar como um
bilhete de identificação para o paciente, visando otimizar a confirmação de presença
e a busca pelo registro na recepção física, eliminando a necessidade de digitar o
protocolo ou CPF.

História de Usuário:
Eu, Fernanda (Recepcionista), desejo que o paciente me apresente um QR Code na
chegada, para que eu possa escanear o código e confirmar a presença e puxar
rapidamente o prontuário sem precisar digitar o protocolo ou CPF, agilizando o
atendimento na recepção.

Critérios de Aceitação:
1. O sistema deve gerar um QR Code único e dinâmico imediatamente após o status do
   agendamento mudar para "Confirmado", e o código deve codificar o Número de
   Protocolo e o ID único do Agendamento.
2. A credencial (com o QR Code) deve ser automaticamente disponibilizada no portal
   do paciente e enviada por meio de notificação via WhatsApp e e-mail.
3. O sistema deve fornecer uma interface administrativa capaz de escanear o QR Code
   via câmera ou scanner e realizar a busca do registro do paciente.
4. Após a leitura bem-sucedida do QR Code na recepção, o sistema deve exibir
   automaticamente os dados do agendamento e permitir que o recepcionista mude o
   status para "Presença Confirmada" com um clique.
5. O QR Code deve ser considerado inválido se a data e o horário do agendamento
   tiverem sido ultrapassados em mais de 1 hora ou se o status do agendamento tiver
   sido alterado para "Cancelado" ou "Falta".
6. A credencial digital ou impressa deve incluir um resumo das instruções específicas
   (documentos, preparo) necessárias para o atendimento, dinamicamente baseadas
   no Tipo de Atendimento.

RF50 – Disparar Lembretes Automáticos de Documentos Pendentes
--------------------------------------------------------------------------------

Descrição:
Implementar um sistema automatizado de notificação para alertar pacientes com
solicitações no status "Pendente" ou "Em Análise" que ainda não enviaram os
documentos obrigatórios (ex: encaminhamento médico). O objetivo é reduzir o tempo
que as solicitações ficam paradas na fila de triagem por falta de documentação,
melhorando a eficiência do processo e a taxa de conversão de agendamentos.

História de Usuário:
Eu, Patrícia (Recepcionista), desejo que o sistema envie lembretes automáticos
aos pacientes sobre documentos pendentes, para que eu não precise checar e
notificar cada caso manualmente, liberando meu tempo para focar na triagem das
solicitações com documentação completa.

Critérios de Aceitação:
1. O sistema deve verificar o status das solicitações diariamente, identificando
   automaticamente todos os registros com status "Pendente" ou "Em Análise" que
   não possuem anexos obrigatórios.
2. O sistema deve disparar uma notificação automática via WhatsApp e e-mail,
   contendo a lista de documentos faltantes e o link seguro para o paciente realizar
   o upload.
3. O primeiro lembrete deve ser enviado 48 horas após o registro inicial da solicitação,
   e os subsequentes devem ser enviados a cada 7 dias, limitados a um total de 3 lembretes.
4. Caso o paciente realize o upload dos documentos faltantes a qualquer momento após
   o envio do primeiro lembrete, o sistema deve interromper imediatamente a sequência
   de lembretes agendados para aquela solicitação.
5. O dashboard administrativo deve exibir um log de comunicação que mostre claramente
   a data e a hora de envio de cada um dos 3 lembretes disparados, permitindo que a
   recepcionista monitore a aderência do paciente.
6. Se o paciente não enviar os documentos obrigatórios após o terceiro e último lembrete,
   o sistema deve automaticamente mudar o status da solicitação para "Arquivado por
   Documentação Pendente" após 24 horas, e notificar o paciente sobre a necessidade
   de uma nova solicitação caso ainda deseje o atendimento.

RF53 – Solicitar Reagendamento de Consulta
--------------------------------------------------------------------------------

Descrição:
Permitir que o paciente solicite a alteração da data ou horário de um agendamento
já confirmado sem precisar cancelar e iniciar um novo processo.

História de Usuário:
Bruno (Paciente) deseja solicitar a troca da data da minha consulta através do portal,
para que eu consiga ajustar o atendimento aos meus imprevistos de forma simples.

Critérios de Aceitação:
1. O botão de "Solicitar Reagendamento" deve aparecer apenas para consultas com status "Confirmado".
2. O sistema deve exibir um calendário com as novas datas e horários disponíveis para
   o mesmo tipo de atendimento.
3. Ao selecionar uma nova data, o sistema deve informar que a alteração está sujeita
   à aprovação da triagem.
4. O sistema deve exigir que o paciente informe o motivo da mudança em um campo de texto breve.
5. Após a solicitação, o status da consulta original deve mudar para "Reagendamento Pendente".
6. O paciente deve receber uma confirmação (e-mail ou notificação no portal) assim que a
   administração aprovar ou sugerir um novo horário.

RF56 – Painel de Privacidade e Gestão de Dados (Self-Service)
--------------------------------------------------------------------------------

Descrição:
Disponibilizar uma área no perfil do utilizador onde ele pode gerir as suas
preferências de privacidade, consultar quais dados o hospital possui e exercer os
seus direitos de titular.

História de Usuário:
Sandra (Paciente) deseja ter um painel centralizado de privacidade, para que possa
desativar comunicações de marketing ou solicitar uma cópia dos meus dados sem
precisar contactar o hospital por telefone.

Critérios de Aceitação:
1. O sistema deve listar todas as finalidades de uso de dados (ex: Notificações de
   Consulta, Pesquisas de Satisfação, Newsletters).
2. Deve permitir que o utilizador ative ou desative permissões opcionais (ex: desativar
   e-mails de marketing, mas manter e-mails de agendamento).
3. Deve conter um botão visível para "Exportar Meus Dados", que gera um arquivo
   (JSON ou CSV) com todas as informações pessoais do utilizador.
4. O sistema deve permitir que o usuário peça a exclusão da sua conta e de seus dados
   diretamente pelo painel.
5. O painel deve exibir a data e hora do último consentimento fornecido pelo utilizador.
6. O sistema deve apresentar um link direto para a Política de Privacidade atualizada da instituição.

RF88 – Permitir Substituição de Documentos Pendentes
--------------------------------------------------------------------------------

Descrição:
Permitir que o paciente faça o upload de uma nova versão de um anexo (como um
encaminhamento ou laudo) diretamente pelo portal, quando o documento anterior for
identificado pela administração como inválido, incorreto ou ilegível.

História de Usuário:
Eu, Rafael (Paciente), desejo substituir o documento que enviei errado ou ilegível
diretamente pelo portal, para que a triagem administrativa possa prosseguir rapidamente
com meu pedido e eu não precise fazer uma nova solicitação.

Critérios de Aceitação:
1. O sistema deve exibir, na área de acompanhamento de status (RF06), a lista de
   documentos anexados à solicitação, destacando aqueles marcados pela administração
   como "Ilegível" ou "Pendente de Correção".
2. Para documentos com status de pendência, um botão "Substituir Documento" deve ser
   visível, permitindo ao paciente carregar um novo arquivo.
3. O sistema deve permitir o upload do novo arquivo, que deve passar imediatamente pelas
   validações de segurança (tipo e tamanho) definidas em RNF82.
4. Após o upload bem-sucedido, o sistema deve automaticamente ocultar o documento antigo
   da visualização e definir o novo anexo com o status "Em Reanálise".
5. O sistema deve registrar a ação de substituição do documento no log de auditoria,
   detalhando qual documento foi substituído, o ID do paciente e o timestamp.
6. O sistema deve enviar uma notificação interna imediata para o dashboard administrativo
   alertando a equipe de triagem que um documento pendente foi corrigido pelo paciente
   e requer uma nova verificação.

RNF45 – Garantir Conformidade Legal e Privacidade (LGPD)
--------------------------------------------------------------------------------

Descrição:
Garantir que todo o tratamento de dados pessoais e, em especial, dos dados sensíveis
de saúde dos pacientes, esteja em estrita conformidade com a Lei Geral de Proteção
de Dados. O sistema deve incorporar mecanismos de segurança, consentimento explícito
e rastreabilidade que assegurem a privacidade, a integridade e a confidencialidade das
informações em todas as etapas do processo.

História de Usuário:
Eu, Clóvis (Setor Jurídico), desejo que o sistema de agendamento seja desenvolvido em
total conformidade com a LGPD (Lei Geral de Proteção de Dados), incluindo políticas de
privacidade e consentimento explícito em cada etapa de coleta de dados sensíveis, para
que o hospital minimize os riscos legais e regulatórios e assegure a rastreabilidade de
todos os acessos às informações dos pacientes.

Critérios de Aceitação:
1. O formulário de solicitação e a tela de resumo devem incluir um checkbox de Consentimento
   Explícito e destacado para o tratamento de dados sensíveis de saúde, sem o qual o envio
   da solicitação será bloqueado.
2. O sistema deve implementar a criptografia de ponta a ponta para todos os dados em trânsito
   e criptografia "at rest" (no banco de dados) para dados sensíveis, como o CPF, resultados
   de exames e histórico de atendimento.
3. O sistema deve manter um Log de Acesso Dedicado que registre o usuário administrativo,
   data, hora e finalidade de toda visualização ou download de documentos anexados ou de
   dados sensíveis de saúde.
4. O portal do paciente deve disponibilizar uma funcionalidade que permita ao titular dos dados
   solicitar a exportação completa de seus dados pessoais (Direito de Acesso) e a
   anonimização/exclusão de dados não essenciais para retenção legal (Direito ao Esquecimento).
5. Em todos os relatórios gerenciais, gráficos e exportações de dados, o sistema deve apresentar
   os dados do paciente de forma pseudonimizada ou anonimizada (ex: usando apenas o número
   de protocolo), exceto se o usuário administrativo possuir um nível de acesso explícito para
   visualização do dado completo.
6. O sistema deve disparar um alerta de segurança imediato para o Setor Jurídico e o Setor de T.I.
   do hospital caso detecte múltiplas tentativas de acesso não autorizado, manipulação de hashes
   de integridade ou qualquer indício de violação de dados.

RNF47 – Garantir Acessibilidade
--------------------------------------------------------------------------------

Descrição:
Garantir que a plataforma atenda aos padrões mínimos de acessibilidade digital, em
conformidade com a Lei Brasileira de Inclusão (LBI) e os critérios do WCAG (Web Content
Accessibility Guidelines) nível AA. O objetivo é assegurar o acesso a pacientes com
deficiência visual, auditiva, física ou cognitiva, promovendo a autonomia no uso do portal
de agendamento e o acesso igualitário aos serviços do hospital.

História de Usuário:
Eu, Laura (Paciente), desejo que o portal de agendamento seja compatível com leitores de
tela e navegação por teclado, para que eu consiga solicitar e acompanhar meu atendimento
com autonomia, garantindo o acesso igualitário aos serviços do hospital.

Critérios de Aceitação:
1. O sistema deve ser totalmente navegável utilizando apenas o teclado (tab-order e atalhos)
   em todas as telas de interação do paciente e do administrador.
2. Todas as imagens, gráficos e elementos interativos devem possuir atributos de texto
   alternativo e rótulos descritivos para serem interpretados corretamente por leitores de tela.
3. As cores utilizadas na interface devem atender aos padrões de contraste mínimo definidos
   pelo WCAG nível AA, garantindo legibilidade para usuários com baixa visão ou daltonismo.
4. O sistema deve ser compatível com as ferramentas de alto contraste e redimensionamento de
   fontes nativas dos navegadores, sem quebrar o layout ou ocultar informações essenciais.
5. A interface deve garantir que as mensagens de erro ou alertas críticos sejam exibidas de forma
   visual e sonora (ou através de recursos de acessibilidade) que não dependam apenas da cor
   para transmitir a informação.
6. O código do sistema deve ser validado por ferramentas automatizadas de acessibilidade para
   garantir que não haja erros críticos ou graves de conformidade.

RNF48 – Garantir Performance de Carga e Estabilidade
--------------------------------------------------------------------------------

Descrição:
Assegurar que o sistema possua performance otimizada e escalável para suportar o alto
volume de tráfego (pico de demanda), garantindo a estabilidade e um tempo de resposta
rápido para todas as operações críticas do paciente (solicitação, acompanhamento) e da
triagem administrativa.

História de Usuário:
Eu, Guilherme (Diretor Administrativo), desejo que o sistema permaneça rápido e estável,
com baixa latência, mesmo durante os picos de demanda, para que a operação e a
experiência do paciente não sejam comprometidas.

Critérios de Aceitação:
1. O tempo de carregamento inicial (First Contentful Paint) de qualquer página principal no
   módulo do paciente e no dashboard administrativo não deve exceder 3 segundos, mesmo
   sob carga normal de usuários.
2. O sistema deve ser submetido a testes de carga que comprovem a capacidade de suportar
   um volume 50% superior ao pico histórico de solicitações, sem que o tempo de resposta das
   consultas ao banco de dados exceda 500ms.
3. O tempo de processamento da Triagem Administrativa e da Realocação de Vagas não deve
   demorar mais que 1 segundo após a ação do usuário.
4. A latência total (tempo de ida e volta) para as chamadas de API de integração com o
   Prontuário Eletrônico não deve exceder 800 ms na média das requisições.
5. As consultas de banco de dados para a geração de Relatórios Gerenciais não devem travar
   as operações ativas da triagem e devem ser executadas em, no máximo, 15 segundos,
   mesmo para grandes volumes de dados.
6. O sistema deve garantir uma taxa de disponibilidade de no mínimo 99%, exceto em janelas
   de manutenção programada.

RNF77 – Otimizar o Peso de Carregamento
--------------------------------------------------------------------------------

Descrição:
O sistema deve ser leve para garantir o funcionamento em ligações de internet limitadas
ou instáveis.

História de Usuário:
Seu Geraldo (Paciente) deseja que as páginas do portal carreguem rápido no meu celular
mesmo quando o sinal 3G estiver fraco dentro do hospital, para que eu consiga ver o meu
horário sem demora.

Critérios de Aceitação:
1. O peso total da página inicial não deve ultrapassar 2MB.
2. Comprimir automaticamente todas as imagens da interface.
3. Utilizar lazy loading para carregar imagens apenas quando visíveis.
4. Minificar ficheiros de código (JS e CSS).
5. Manter funcionalidade legível em ligações simuladas de 3G.
6. Tempo de resposta inicial do servidor (TTFB) inferior a 500ms.

RNF81 – Implementar o Salvamento Automático de Formulários
--------------------------------------------------------------------------------

Descrição:
O sistema deve guardar o progresso de preenchimento de formulários longos para
evitar perda de dados.

História de Usuário:
Maria (Paciente) deseja que o sistema guarde automaticamente as informações que
estou a preencher no meu histórico clínico, para que eu não perca tudo se a minha
internet cair.

Critérios de Aceitação:
1. O salvamento deve ocorrer automaticamente a cada 30 segundos de inatividade.
2. Deve exibir um aviso discreto de "Rascunho guardado" no canto do ecrã.
3. O sistema deve restaurar os dados guardados assim que o utilizador voltar à página.
4. Os dados temporários devem ser apagados apenas após o envio bem-sucedido do formulário.
5. O processo de salvamento não deve causar lentidão na digitação do utilizador.
6. Deve funcionar mesmo que o utilizador feche o navegador acidentalmente.

RNF84 – Disponibilizar Ajuda Contextual por Ecrã
--------------------------------------------------------------------------------

Descrição:
O sistema deve fornecer explicações breves sobre termos técnicos ou campos complexos
em cada funcionalidade.

História de Usuário:
Seu Geraldo (Paciente) deseja ver uma pequena explicação ao passar o rato sobre termos
médicos, para que eu entenda exatamente o que devo preencher ou ler.

Critérios de Aceitação:
1. Incluir ícones de interrogação (?) ao lado de campos que exijam dados específicos.
2. Exibir o texto de ajuda (tooltip) ao passar o cursor ou tocar no ícone.
3. O texto de ajuda deve ser breve e não ultrapassar 140 caracteres.
4. Incluir um link para a central de ajuda ou FAQ no rodapé de cada módulo.
5. Garantir que os leitores de ecrã consigam ler o conteúdo da ajuda (acessibilidade).
6. Permitir que o administrador edite os textos de ajuda através do painel de gestão.
