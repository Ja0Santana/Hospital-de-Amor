RF10 – Incentivar Fidelização
--------------------------------------------------------------------------------

Descrição:
O sistema poderá oferecer benefícios simbólicos ou concretos, assim o usuário
conseguirá gastar os pontos de fidelização.

História de Usuário:
Eu, Tiago (Doador e voluntário) desejo conseguir converter minhas contribuições
em pontos de reconhecimento para resgatar benefícios e fortalecer meu vínculo
com o hospital.

Critérios de Aceitação:
1. O sistema deve atribuir pontuações automaticamente para ações como doações
   confirmadas.
2. O sistema deve listar benefícios (simbólicos ou concretos) disponíveis para troca
   conforme o saldo de pontos.
3. O usuário deve conseguir visualizar seu saldo atual e o histórico de benefícios já
   resgatados.
4. O sistema deve definir uma validade para os pontos acumulados (ex: 12 ou 24
   meses), notificando o usuário via e-mail (RF07) com 30 dias de antecedência sobre
   qualquer saldo prestes a expirar.
5. O sistema deve possuir níveis de fidelidade (ex: Bronze, Prata, Ouro) baseados
   no total de pontos acumulados na vida do doador, desbloqueando selos simbólicos
   exclusivos no perfil do usuário para reconhecimento público no "Mural de
   Gratidão".
6. No momento da troca por um benefício, o sistema deve verificar em tempo real
   se o saldo de pontos é suficiente, realizar o débito imediato após a confirmação e
   gerar um voucher digital ou comprovante de resgate com QR Code para controle
   administrativo.

RF11 – Ver Histórico de Doações
--------------------------------------------------------------------------------

Descrição:
O sistema deve exibir um histórico de doações do usuário doador dentro de um
certo período especificado.

História de Usuário:
Eu, Carlos (Doador) quero visualizar minhas contribuições anteriores para ter o
controle financeiro das minhas doações ao hospital.

Critérios de Aceitação:
1. Listagem cronológica contendo data, valor e método de pagamento.
2. Disponibilizar filtro de busca por período especificado (mês/ano).
3. O sistema deve permitir que o doador utilize um campo de busca para encontrar
   doações específicas pelo ID da Transação (E2E ID) ou Hash da Transação.
4. O sistema deve exibir, no topo da lista, o valor total acumulado das doações de
   acordo com os filtros aplicados (mês/ano), facilitando a visualização do montante
   doado no período selecionado.
5. A listagem deve indicar claramente o status de cada transação (ex:
   "Processando", "Confirmada" ou "Estornada"), garantindo que apenas doações
   efetivamente liquidadas componham o resumo financeiro.
6. O sistema deve oferecer um botão específico para gerar um relatório
   consolidado de todas as doações realizadas no ano fiscal anterior, formatado para
   auxiliar o doador em sua declaração de Imposto de Renda.

RF19 – Convidar Novos Doadores
--------------------------------------------------------------------------------

Descrição:
O usuário doador deve poder indicar pessoas para doarem, assim incentivando a
novas pessoas a ajudarem.

História de Usuário:
Eu, Mariana (Doadora), desejo enviar convites para meus conhecidos, para
incentivar novos doadores a ajudarem a instituição.

Critérios de Aceitação:
1. Gerar link ou código único de indicação para o usuário logado.
2. Integração com botões de compartilhamento (WhatsApp, E-mail ou Redes Sociais).
3. Registro automático da indicação para fins de pontuação no programa de
   fidelidade.
4. O link gerado deve direcionar o convidado para uma página de boas-vindas
   personalizada, contendo o nome da pessoa que indicou (ex: "Mariana convidou
   você para apoiar o Hospital de Amor") e um resumo da missão institucional.
5. O sistema deve disparar uma notificação automática para o doador original
   (RF07) assim que um de seus convidados realizar a primeira doação confirmada,
   reforçando o impacto positivo da sua indicação.
6. Para garantir o uso ético da ferramenta, o sistema deve implementar uma trava
   de segurança que limite o número de convites enviados por e-mail via plataforma
   em um curto período, incentivando o compartilhamento manual via link direto em
   redes sociais.

RF21 – Pagar por Pix
--------------------------------------------------------------------------------

Descrição:
Permitir que o usuário doador realize pagamentos via Pix.

História de Usuário:
Eu, Carolina (Doadora), desejo pagar minha doação usando a chave Pix, para que
a transação seja instantânea e o hospital receba o valor rapidamente.

Critérios de Aceitação:
1. O sistema deve gerar um QR Code e um código "Copia e Cola" (Pix Copia e Cola)
   para o valor da doação.
2. O sistema deve exibir um cronômetro de validade para o Pix gerado, informando
   o prazo para o pagamento.
3. Após a confirmação bancária, o sistema deve registrar a doação e notificar o
   doador imediatamente.
4. O sistema deve permitir que o doador escolha valores pré-definidos ou insira um
   valor customizado antes de gerar o Pix.
5. Caso o prazo do cronômetro expire sem pagamento, o QR Code deve ser
   invalidado automaticamente no sistema.
6. O sistema deve armazenar o ID da transação (E2E ID) para fins de conciliação
   financeira no módulo institucional.

RF22 – Pagar por Cartão
--------------------------------------------------------------------------------

Descrição:
Permitir que o usuário doador realize pagamentos via Cartão de Crédito/Débito.

História de Usuário:
Eu, Ricardo (Doador), desejo pagar minha doação usando meu cartão de crédito,
para ter flexibilidade no pagamento e garantir que a contribuição seja concluída
imediatamente.

Critérios de Aceitação:
1. O sistema deve integrar-se a um gateway de pagamento (ex: MercadoPago,
   Stripe, etc.) para processar transações com cartão de crédito e débito.
2. O sistema deve exigir campos padrão de cartão (número, nome impresso,
   validade e código de segurança CVV) e validá-los antes da tentativa de cobrança.
3. Após a autorização da operadora, o sistema deve registrar a doação como
   "Confirmada" e enviar o comprovante ao doador.
4. O sistema deve garantir a segurança dos dados utilizando criptografia
   (SSL/TLS) e seguir normas de segurança (PCI-DSS).
5. Em caso de recusa da operadora, o sistema deve exibir uma mensagem clara
   informando o motivo (ex: saldo insuficiente ou dados incorretos).
6. Deve haver uma opção para o doador selecionar se a doação via cartão será
   "Única" ou "Mensal" (recorrente).

RF23 – Pagar por Boleto
--------------------------------------------------------------------------------

Descrição:
O sistema deve permitir que o usuário doador realize pagamentos via Boleto.

História de Usuário:
Eu, Pedro (Doador), desejo gerar um boleto para minha doação, para poder pagar
em uma data posterior ou em caixas físicos (banco, lotérica), utilizando um meio de
pagamento amplamente aceito.

Critérios de Aceitação:
1. O sistema deve gerar um Boleto Bancário em formato PDF, com código de barras e
   informações de pagamento, para que o doador possa imprimir ou salvar.
2. O sistema deve enviar o PDF do boleto para o e-mail cadastrado do doador, além
   de exibi-lo na tela.
3. A doação deve ser registrada com o status "Aguardando Pagamento" e só deve ser
   atualizada para "Confirmada" após a compensação bancária.
4. O boleto gerado deve ter uma data de vencimento padrão de 3 dias úteis após a
   emissão.
5. O sistema deve permitir a visualização do código de barras numerado (linha
   digitável) diretamente na tela para facilitar o pagamento via app bancário.
6. Caso o boleto vença, o sistema deve atualizar o status da doação para "Expirado"
   e permitir que o usuário gere um novo se desejar.

RF24 – Pagar por Criptomoedas
--------------------------------------------------------------------------------

Descrição:
Permitir que o usuário doador realize pagamentos via criptomoedas.

História de Usuário:
Eu, João (Doador Moderno), desejo ter a opção de doar usando criptomoedas (como
Bitcoin ou Ethereum), para utilizar um meio de pagamento seguro e descentralizado
ao apoiar o hospital.

Critérios de Aceitação:
1. O sistema deve gerar um endereço de carteira (wallet address) dinâmico e um
   QR Code para recebimento da criptomoeda selecionada.
2. O sistema deve exibir a taxa de câmbio (conversão para BRL) em tempo real e o
   valor exato a ser transferido, incluindo a rede/network correta.
3. A doação deve ser registrada como "Confirmada" após a validação da transação
   na blockchain.
4. O sistema deve informar ao usuário a rede (network) correta para a transferência
   para evitar perda de fundos (ex: Rede ERC-20 ou Bitcoin).
5. O valor da cotação deve ser garantido por um período limitado (ex: 15 minutos)
   enquanto a transação é processada.
6. O sistema deve registrar a "Hash da Transação" (TXID) no histórico de doações
   para permitir auditoria externa.

RF55 – Enviar Mensagem de Apoio
--------------------------------------------------------------------------------

Descrição:
Permitir que o doador escreva uma breve mensagem de incentivo para ser exibida
internamente no hospital.

História de Usuário:
Eu, Lúcia (Doadora) desejo enviar uma mensagem de carinho para a equipe e
pacientes, para oferecer apoio moral sem envolver dinheiro ou doação de itens.

Critérios de Aceitação:
1. O sistema deve exibir um campo de texto simples para a escrita da mensagem.
2. Deve haver um limite máximo de 300 caracteres para garantir que a mensagem
   seja breve.
3. O doador deve clicar num botão "Enviar Mensagem" para finalizar.
4. O sistema deve exibir um aviso de "Mensagem enviada com sucesso" após o clique.
5. Deve haver uma caixa de seleção para o doador autorizar (ou não) a exibição do
   seu primeiro nome junto à mensagem.
6. A mensagem deve ser salva numa lista simples para conferência da administração.

RF60 – Mural de Transparência
--------------------------------------------------------------------------------

Descrição:
Painel público ou restrito para prestação de contas sobre o uso das doações.

História de Usuário:
Eu, João (Cidadão/Doador) desejo visualizar gráficos sobre como o dinheiro das
doações está sendo investido, para confiar ainda mais na instituição.

Critérios de Aceitação:
1. O sistema deve exibir gráficos simplificados de "Entradas" (doações) vs "Saídas"
   (investimentos).
2. Os dados devem ser consolidados mensalmente para garantir clareza na prestação
   de contas.
3. O mural deve destacar projetos específicos que foram concluídos com a ajuda das
   doações (ex: nova ala).
4. O mural deve exibir o carimbo de data e hora da última atualização dos dados
   financeiros, garantindo aos doadores que a informação é recente e confiável.
5. Os dados de doadores individuais não devem ser expostos, mantendo o
   anonimato conforme a LGPD.
6. O administrador deve ser o responsável por validar e publicar os dados que
   aparecerão no mural.

RF66 – Portal de Patrocínio Corporativo
--------------------------------------------------------------------------------

Descrição:
Área exclusiva para empresas (PJ) realizarem doações de grande porte ou
patrocínio de alas hospitalares.

História de Usuário:
Eu, Dra. Fernanda Lins (Diretora de Responsabilidade Social da Empresa), desejo
cadastrar minha empresa como patrocinadora oficial do Hospital de Amor, gerenciar
os representantes autorizados da nossa organização e acompanhar o impacto das
nossas contribuições, para que possamos cumprir nossas obrigações de prestação de
contas de RSC, obter os documentos fiscais necessários para dedução junto à
Receita Federal e fortalecer a visibilidade institucional da marca nos canais do
hospital.

Critérios de Aceitação:
1. O sistema deve permitir o upload do logo da empresa para exibição na área de
   "Parceiros".
2. Deve gerar relatórios específicos de impacto social para prestação de contas de
   responsabilidade social corporativa.
3. Emitir notas fiscais de doação específicas para pessoas jurídicas.
4. O perfil da empresa deve permitir o cadastro de múltiplos representantes (ex:
   Marketing e RH) com diferentes níveis de permissão.
5. O sistema deve disponibilizar para download um "Selo de Responsabilidade Social"
   digital, com validade anual, para que a empresa utilize em seus próprios canais
   de comunicação.
6. O sistema deve enviar lembretes automáticos para os representantes da empresa
   60 dias antes do vencimento de um contrato de patrocínio ou cota anual.
